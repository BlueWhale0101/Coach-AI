Fitness Logging & Analysis System

Purpose

This system captures fitness, nutrition, recovery, and coaching information from conversational interactions with ChatGPT.

The primary design goal is:

Logging must remain conversational.

Storage must be structured.

Raw text must always be preserved.

The database is the source of truth.

The GPT acts as:

* Coach
* Data entry assistant
* Analysis engine

The web application exists primarily as an administrative and debugging interface.

⸻

Design Principles

1. Raw Text is Sacred

Every user entry must be preserved exactly as originally entered.

No structured extraction process may replace or overwrite the original text.

The original text remains the authoritative record.

⸻

2. Structured Data is Secondary

Structured records exist to support:

* searching
* reporting
* trend analysis
* coaching

Structured records may be incomplete.

Structured records may be corrected later.

Raw text remains authoritative.

⸻

3. Facts vs Estimates vs Interpretation

The system separates:

Facts

Explicitly stated by the user.

Examples:

* bodyweight
* workout performed
* foods consumed
* low energy
* high hunger

Estimates

Generated by GPT.

Examples:

* calorie ranges
* protein ranges
* workout duration ranges
* training load estimates

Estimates must be stored as ranges whenever practical.

Interpretation

Coaching conclusions.

Examples:

* possible under-fueling
* excessive training load
* recovery concerns

Interpretations must not be stored as factual user data.

Interpretations belong in coach observations and summaries.

⸻

4. Null is Better than Wrong

If information is not explicitly available:

Store NULL.

Do not invent values.

Example:

User says:

“I am tired.”

Valid:

* recovery event
* description: “I am tired”

Invalid unless explicitly stated:

* sleep_quality = poor
* stress = high
* soreness = moderate

⸻

5. Flexible First, Normalize Later

The first implementation should avoid premature over-normalization.

Rather than creating separate tables for meals, workouts, weigh-ins, recovery notes, sleep notes, and symptoms immediately, v0.1 uses one flexible child table:

fitness_events

Specialized tables may be added later if real usage proves they are needed.

⸻

6. Scalability

All read operations must support:

- limit
- offset

Even if the initial dataset is small.

This avoids future interface changes as historical data grows.

⸻

High-Level Architecture

ChatGPT Coach
      |
GPT Actions
      |
Supabase API / Edge Functions
      |
PostgreSQL
      |
GitHub Pages Admin App

The database is the system of record.

GPT is stateless between conversations.

All durable information must be stored in the database.

⸻

Core Relationship Model

log_entries
    |
    +-— fitness_events
    |
    +-— coach_observations
daily_summaries
import_batches

A single user message may generate multiple child records.

Example user message:

Weighed 198.0. Hard HYROX session. Brekky eggs and mushrooms.

Produces:

1 log_entry
3 fitness_events:
- weigh_in
- workout
- meal

All child records reference the original log entry.

v0.1 Tables

1. log_entries

Purpose:

Immutable source record for user-submitted natural-language logs.

Stores:

* original raw text
* occurrence date/time
* source
* import metadata
* review metadata

Rules:

* Never overwrite raw_text
* One user message creates one log_entry
* A log_entry may produce zero, one, or many fitness_events
* Raw text remains authoritative

Draft schema:

create table log_entries (
  id uuid primary key default gen_random_uuid(),
  occurred_date date not null,
  occurred_time time null,
  raw_text text not null,
  source text not null check (
    source in (
      ‘gpt_action’,
      ‘chat_backfill’,
      ‘web_admin’,
      ‘manual’
    )
  ),
  import_batch_id uuid null,
  extraction_status text not null default ‘pending’ check (
    extraction_status in (
      ‘pending’,
      ‘extracted’,
      ‘partially_extracted’,
      ‘failed’,
      ‘skipped’
    )
  ),
  extraction_confidence numeric(3,2) null,
  needs_review boolean not null default false,
  created_at timestamptz not null default now(),
  updated_at timestamptz not null default now()
);

⸻

2. fitness_events

Purpose:

Flexible structured child records extracted from log_entries.

Event types may include:

* meal
* workout
* weigh_in
* recovery
* sleep
* note
* social_event
* alcohol
* symptom

Rules:

* Every fitness_event must reference a log_entry
* Each event should preserve the specific source text span that produced it
* Facts, estimates, and interpretations are separated
* Missing information remains null or absent from JSON
* Do not store GPT guesses as facts

Draft schema:

create table fitness_events (
  id uuid primary key default gen_random_uuid(),
  log_entry_id uuid not null references log_entries(id) on delete cascade,
  occurred_date date not null,
  occurred_time time null,
  event_type text not null check (
    event_type in (
      ‘meal’,
      ‘workout’,
      ‘weigh_in’,
      ‘recovery’,
      ‘sleep’,
      ‘note’,
      ‘social_event’,
      ‘alcohol’,
      ‘symptom’
    )
  ),
  title text null,
  description text not null,
  source_text_span text null,
  facts jsonb not null default ‘{}’::jsonb,
  estimates jsonb not null default ‘{}’::jsonb,
  interpretations jsonb not null default ‘{}’::jsonb,
  extraction_confidence numeric(3,2) null,
  estimate_confidence numeric(3,2) null,
  needs_review boolean not null default false,
  created_at timestamptz not null default now(),
  updated_at timestamptz not null default now()
);

Example meal event:

{
  “event_type”: “meal”,
  “description”: “3 eggs, mushrooms, 1/2 English muffin”,
  “source_text_span”: “Brekky: 3 eggs, mushrooms, 1/2 English muffin”,
  “facts”: {
    “meal_label”: “breakfast”
  },
  “estimates”: {
    “calories_low”: 350,
    “calories_high”: 500,
    “protein_g_low”: 22,
    “protein_g_high”: 32
  },
  “interpretations”: {},
  “extraction_confidence”: 0.95,
  “estimate_confidence”: 0.75,
  “needs_review”: false
}

Example weigh-in event:

{
  “event_type”: “weigh_in”,
  “description”: “Weighed 198.0”,
  “source_text_span”: “Weighed 198.0”,
  “facts”: {
    “weight_value”: 198.0,
    “weight_unit”: “lb”
  },
  “estimates”: {},
  “interpretations”: {},
  “extraction_confidence”: 1.0,
  “estimate_confidence”: null,
  “needs_review”: false
}

Example recovery event:

{
  “event_type”: “recovery”,
  “description”: “Low energy and hungry”,
  “source_text_span”: “I’m low energy and hungry”,
  “facts”: {
    “energy_level”: “low”,
    “hunger_level”: “high”,
    “explicitly_mentioned_fields”: [“energy_level”, “hunger_level”]
  },
  “estimates”: {},
  “interpretations”: {},
  “extraction_confidence”: 0.95,
  “estimate_confidence”: null,
  “needs_review”: false
}

3. coach_observations

Purpose:

Stores coaching analysis, interpretation, and recommendations.

Coach observations are not automatically created during add_log_entry.

Logging and coaching are intentionally separated.

The system should not create large numbers of AI-generated records during normal logging.

Coach observations may be created later by dedicated analysis workflows.

This table is not factual user data.

Rules:

* Observations may reference one or more log entries
* Observations may reference one or more fitness events
* Interpretations should not pollute factual event fields
* Observations may be regenerated or superseded

Draft schema:

create table coach_observations (
  id uuid primary key default gen_random_uuid(),
  log_entry_id uuid null references log_entries(id) on delete set null,
  occurred_date date not null,
  observation_type text not null check (
    observation_type in (
      ‘fueling’,
      ‘recovery’,
      ‘training_load’,
      ‘weight_trend’,
      ‘behavior’,
      ‘risk’,
      ‘positive_reinforcement’,
      ‘other’
    )
  ),
  observation_text text not null,
  evidence_log_entry_ids uuid[] null,
  evidence_fitness_event_ids uuid[] null,
  confidence numeric(3,2) null,
  actionable boolean not null default false,
  created_at timestamptz not null default now(),
  updated_at timestamptz not null default now()
);

⸻

4. daily_summaries

Purpose:

Derived daily overview for trend review and coaching continuity.

Rules:

* Derived data
* Regeneratable
* Not source of truth
* Should reference source log entries and/or events when practical

Draft schema:

create table daily_summaries (
  id uuid primary key default gen_random_uuid(),
  summary_date date not null unique,
  bodyweight_value numeric(5,1) null,

      ‘light’,

  sleep_summary text null,
  coach_summary text null,
  flags jsonb not null default ‘[]’::jsonb,
  source_log_entry_ids uuid[] null,
  source_fitness_event_ids uuid[] null,
  generated_by text not null default ‘gpt’,
  needs_review boolean not null default false,
  created_at timestamptz not null default now(),
  updated_at timestamptz not null default now()

5. import_batches

Purpose:

Tracks bulk imports and backfills.

Examples:

* historical chat imports
* image-based weight logs
* manual JSON imports
* admin uploads

Rules:

* Supports auditability
* Imported log_entries may reference an import batch
* Import failures should be visible in the admin app

Draft schema:

create table import_batches (

GPT Action Design

v0.1 should expose only a small number of actions.

Primary action:

add_log_entry

The user should not need to choose whether something is a meal, workout, recovery note, or weigh-in.

The GPT should accept a conversational log, preserve the raw text, and extract one or more fitness_events.


# v0.1 Action Set

WRITE

- add_log_entry
- update_event

READ

- get_logs
- search_events
- get_day_summary

ANALYSIS SUPPORT

- get_trends

These actions are considered the stable v0.1 interface contract.

Avoid exposing separate actions such as:

add_meal
add_workout
add_weigh_in
add_recovery_note

Those may be added later if needed, but they increase user friction early.

⸻

add_log_entry Concept

Input:

* raw_text
* occurred_date
* occurred_time optional
* source
* extracted fitness_events optional

Output:

* created log_entry
* created fitness_events
* review flags

Behavior:

1. Store raw_text exactly.
2. Create one log_entries row.
3. Extract zero or more fitness_events.
4. Store source_text_span for each event.
5. Store facts, estimates, and interpretations separately.
6. Mark uncertain items needs_review.

⸻

Historical Chat Import

Future workflow:

1. Export historical chat.
2. Convert to machine-readable chunks.
3. Create import_batch.
4. For each message, create log_entry.
5. Extract fitness_events.
6. Review low-confidence records.
7. Generate daily_summaries.

⸻

Image-Based Weight Log Import

Future workflow:

1. Upload image.
2. GPT reads visible weight log entries.
3. For each entry, call add_log_entry or import workflow.
4. Create weigh_in fitness_events.
5. Attach import_batch metadata if processed as bulk import.

⸻

Future Normalized Tables

Not part of v0.1.

Potential future tables:

* meals
* workouts
* weigh_ins
* recovery_notes
* sleep_notes
* symptoms
* alcohol_logs

These should only be added if the flexible fitness_events model becomes limiting.

Until then, v0.1 uses:

fitness_events.facts
fitness_events.estimates
fitness_events.interpretations

as the structured data layer.

# Logging Model

The system uses automatic conversational logging.

The user is not expected to explicitly choose between:

- meal
- workout
- weigh_in
- recovery
- sleep

Instead, the GPT determines whether fitness-relevant information should be logged.

Examples:

User:
“198.0 today.”

GPT:
Creates a log entry and weigh_in fitness event.

User:
“Morning.”

GPT:
No logging action required.

The primary goal is minimizing user logging friction.

# Event Identity

fitness_event IDs are durable.

Events may be updated through update_event.

Event IDs should not change during the lifetime of a record.

Future versions may add revision history, but v0.1 stores only the current version of each event.
